# Suricata IDS Integration — Elastic Cyber Lab
 ### Summary

As part of the Elastic Cyber Lab, I integrated Suricata IDS to provide network-level visibility alongside host-based telemetry.
The objective was to detect internal reconnaissance and attack activity and centralize logs in the Elastic Stack for investigation and detection engineering.

This completed the lab by adding network detection capabilities to the SOC environment.

---

## Lab Environment

I deployed four virtual machines:
```text
Machine	Purpose

Elastic Server :	Elasticsearch & Kibana SIEM

Suricata NIDS :	Network intrusion detection sensor

Kali Linux : Attacker system

Metasploitable 2 :	Vulnerable target
```
---

## Network Architecture

I designed an isolated internal attack network in VirtualBox:
```text
Attack_net (Internal Network)
Used for attacker ↔ target traffic
```
---
```text
NAT Adapter
Used for management and updates
```
Suricata was connected to both networks and configured in promiscuous mode to monitor mirrored traffic.

---

## Suricata Deployment

I configured Suricata in IDS (passive monitoring) mode using AF_PACKET for high-speed packet capture.

Key configurations included:

Disabled IPS/NFQUEUE features

Enabled multi-protocol logging

Enabled EVE JSON output for Elastic ingestion

Monitored the attack network interface

Suricata successfully captured:

DNS traffic

HTTP requests

TLS handshakes

SSH sessions

Network flows

Security alerts

---

## Elastic Stack Integration

To centralize network telemetry:

I enabled the Filebeat Suricata module

Parsed Suricata EVE logs

Forwarded structured events to Elasticsearch

Loaded prebuilt Kibana dashboards

This allowed Suricata alerts and network metadata to be searchable alongside endpoint telemetry.

---

## Attack Simulation & Validation

To validate the deployment, I simulated attacker behavior from Kali Linux.

Activities performed:

Network discovery scans

Service enumeration

Target fingerprinting

Suricata successfully detected reconnaissance activity and generated alerts tied to:

Source IP (attacker)

Destination IP (target)

Scanned ports and services

HTTP requests generated by scanning scripts

All events were visible in Kibana and indexed in Elasticsearch.

---

## Investigation in Kibana

Using Kibana Discover and dashboards, I was able to:

Trace attacker activity across network logs

Inspect Suricata alert metadata

Correlate scan behavior with target services

Visualize traffic patterns and protocol usage

This confirmed end-to-end visibility from packet capture → detection → ingestion → investigation.

The lab successfully evolved from endpoint monitoring to a hybrid detection environment combining:
